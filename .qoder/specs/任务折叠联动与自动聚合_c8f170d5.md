# 任务折叠联动与自动聚合实现方案

## 架构决策摘要

| 决策点       | 选择                                            | 理由                                                                         |
| ------------ | ----------------------------------------------- | ---------------------------------------------------------------------------- |
| Schema 变更  | **不修改**                                      | `Task.duration`（工作日）可作为工时权重，`ViewState.collapsedTaskIds` 已存在 |
| 聚合计算时机 | **Command 层持久化 + Assembly 层渲染兜底**      | Command 保证 undo/redo 一致性；Assembly 兜底处理拖拽中间态                   |
| 折叠状态管理 | 复用 `setViewStateCommand` + `collapsedTaskIds` | 已有完整管道，仅需添加 UI 控件                                               |
| 折叠 UI 交互 | `setViewStateCommand`（非 undoable 争议可接受） | 折叠是离散操作，保留在 undo 栈中是合理行为                                   |

## 核心设计

### 数据流

```
用户编辑子任务 → updateTaskWithRollupCommand → Command.apply()
  ├─ 更新子任务字段
  └─ computeCascadeRollup() 后序遍历
       ├─ 更新父任务 {start, end, duration, progress}
       └─ 更新祖父任务 {start, end, duration, progress}
           └─ ...直到根节点

用户拖拽子任务 → setState(实时更新) → assembleScene()
  └─ toTaskRow() 中注入 rollup 值覆盖 summary 的 start/end/progress
      （仅用于 Canvas 渲染，不持久化）
```

### 加权进度算法

```
weight(child) = child.isSummary ? rollup[child.id].duration : child.duration
parent.progress = Σ(child.progress × weight(child)) / Σ(weight(child))
特殊情况：所有子任务 progress === 100 → parent.progress = 100
除零保护：Σ(weight) === 0 → 简单算术平均
```

### 时间范围推导

```
parent.start = min(children.start)
parent.end = max(children.end)
parent.duration = Σ(children.duration)（工时之和）
```

## 实施步骤

### Step 1: 新建摘要聚合纯函数模块

**新建文件**: `apps/web/src/lib/summary.ts`

- **`computeRollup(children: Task[]): {start, end, duration, progress}`**
  - 对一组子任务计算聚合值
  - `start` = min, `end` = max, `duration` = sum, `progress` = weighted avg
  - 权重使用 `duration`（如子任务是 summary 则使用其已 rollup 的 duration）

- **`computeCascadeRollup(tasks: Task[], changedTaskId: string): Array<{id, patch: Partial<Task>}>`**
  - 构建 `childrenOf` 索引
  - 后序遍历（叶子优先），用 Map 存储每个 summary 节点的 rollup 结果
  - 沿 `parentId` 链向上冒泡，直到根节点或值不再变化
  - 返回所有需要更新的祖先 `{id, patch}` 列表
  - 复杂度 O(n)，n = 任务总数

- **`isSummaryTask(taskId: string, tasks: Task[]): boolean`**
  - 判断任务是否有子节点

- 无副作用纯函数，易于单元测试

### Step 2: 新增 updateTaskWithRollupCommand

**修改文件**: `apps/web/src/store/useProjectStore.ts`

- 新增 `updateTaskWithRollupCommand(taskId: string, patch: Partial<Task>): Command`
  - `apply(file)`:
    1. 将 patch 应用到目标任务
    2. 调用 `computeCascadeRollup(updatedTasks, taskId)` 获取祖先变更
    3. 将所有祖先 patch 一次性应用到 tasks 数组
    4. 捕获所有被修改任务的旧值（用于 invert）
  - `invert(file)`: 恢复所有被修改任务（包括祖先）的旧值
  - label: `更新任务(含汇总)`

- **关键**: `apply` 在闭包内完成所有计算，确保原子性

## 风险与对策

| 风险                               | 严重度 | 对策                                                                                       |
| ---------------------------------- | ------ | ------------------------------------------------------------------------------------------ |
| 拖拽高频触发 rollup 导致卡顿       | 低     | rollup 是 O(n) 纯计算，100 任务 < 1ms；若需优化可 throttle                                 |
| E2E 截图回归                       | 高     | Step 12 专门处理基线更新                                                                   |
| 深层树（10+ 级）冒泡链             | 低     | 实际项目极少超过 5 级；后序遍历一次处理所有层级                                            |
| Command 的 invert 需捕获多任务旧值 | 中     | 在 apply 内用 Map 收集所有修改前的旧值，invert 精确还原                                    |
| 删除子任务后父级仍是 summary       | 低     | `deleteTaskCommand` 级联删除所有后代；若仍有子节点则 rollup 正确计算；若无子节点则变为叶子 |
| 拖拽 summary bar 的行为            | 中     | hitTest 中 summary 行返回 `'empty'`，禁止拖拽                                              |

## 被否决的备选方案

### 方案 B：添加 `effort` 字段到 Schema

- **理由**：增加 schema 迁移复杂度，`duration`（工作日）已能充分表达"工时"概念，与 MS Project 一致

### 方案 C：仅在 Assembly 层计算 rollup，不持久化

- **理由**：TaskTable 直接读 `file.tasks`，如果不持久化则 TaskTable 显示旧值；CPM 关键路径计算也需要正确的 summary 时间
- **采用混合方案**：Command 持久化为主，Assembly 注入为辅（仅处理拖拽中间态）

### 方案 D：在 dispatch() 中添加 post-dispatch hook

- **理由**：增加 store 复杂度，破坏 Command 模式的显式性；专用 Command 更清晰
